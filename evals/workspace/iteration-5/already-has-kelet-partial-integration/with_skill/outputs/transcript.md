# Eval Transcript — already-has-kelet-partial-integration

**Prompt:** "I already have kelet installed but I'm not seeing sessions in the console"

---

## Diagnosis

### What was found

The app is a FastAPI + pydantic-ai docs assistant. Kelet is listed as a dependency in `pyproject.toml` (`kelet>=1.3.0`), but the integration was incomplete:

1. **`kelet.configure()` missing** — `app/main.py` had no `import kelet` and no `kelet.configure()` call at startup. Without this, the SDK never initializes and no traces are exported.

2. **`kelet.agentic_session()` missing** — `src/routers/chat.py` had no session wrapping around the pydantic-ai agent run. Each incoming `/chat` POST triggers `chat_agent.iter(...)` which makes one or more LLM calls, but without `agentic_session()`, each LLM call appears as a separate unlinked trace in the console — there is no grouping into a logical "conversation."

### Why sessions were not appearing

The SKILL.md notes that pydantic-ai is a supported framework (auto-instrumented for trace capture), but the app also **owns the session loop** — `session.session_id` is a Redis-backed UUID generated per conversation in `src/cache/__init__.py`. This means Kelet cannot infer the session boundary on its own: the framework doesn't propagate this ID into its spans. `agentic_session(session_id=...)` is required to associate all LLM calls in a conversation under one session.

### Session ID evaluation

`session.session_id` is a UUID-per-conversation generated by `cache.create_session()` and stored in Redis with a TTL. It changes at each new conversation and is not a stable user identifier. This is the correct mapping — no mismatch.

### Deployment

Kubernetes (Helm chart). `KELET_API_KEY` is injected from a K8s Secret; `KELET_PROJECT` is injected from a ConfigMap. Both env vars are already wired in `k8s/charts/docs-ai/templates/deployment.yaml`. No changes needed to deployment config.

---

## What was fixed

### `app/main.py`
- Added `import kelet`
- Added `kelet.configure()` at module level (after imports, before app creation), reading `KELET_API_KEY` and `KELET_PROJECT` from env

### `src/routers/chat.py`
- Added `import kelet`
- Wrapped the `chat_agent.iter(...)` call inside `async with kelet.agentic_session(session_id=session.session_id):` in `_run_agent_stream`
- The entire streaming generator body (including the `[DONE]` sentinel path) remains inside the session context — the `agentic_session` context manager exits only after all streaming is complete, which is correct
- Added `await kelet.signal("EVENT", "LABEL", session_id=..., trigger_name="agent-stream-error", score=0.0)` in the exception handler to surface agent errors as signals

### What was skipped

- **Stateless endpoint (`GET /chat`)**: This endpoint has no session concept — it's a one-shot query for curl/kelet skill usage. No `agentic_session()` wrapping is appropriate here; individual traces will still appear.
- **Frontend signals**: No frontend was in scope (pure API service). No `VoteFeedback` or `useFeedbackState` proposed.
- **Synthetic evaluators**: Not configured in this pass — the skill would normally generate a deeplink at Checkpoint 2, but this eval is focused on the missing-sessions diagnosis.
